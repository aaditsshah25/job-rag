\section*{Abstract}
\addcontentsline{toc}{section}{Abstract}

JobMatch AI is a full-stack generative AI application that applies retrieval-augmented generation (RAG) to job search. Given a candidate's resume and preferences, the system retrieves relevant job postings from a Pinecone vector store, reranks them using profile-aware scoring, and passes the top candidates to the Gemma language model, which produces a personalised, grounded recommendation response. Beyond job matching, every downstream action in the workflow is AI-powered: cover letter generation, resume tailoring, keyword-gap analysis, and recruiter email drafting are all handled by Gemma with the relevant job description and resume as context.

The system was evaluated across 19 runs using 10 distinct user profiles. Key observed results include Precision@5 = 1.0, NDCG@10 = 0.69, Faithfulness = 1.0, and Format Consistency = 0.75. Faithfulness = 1.0 indicates that, in this evaluation, every recommended job was drawn from the retrieved candidate set, providing strong evidence that the pipeline avoids hallucinated postings. These results are preliminary due to the small sample size, and several metrics rely on proxy relevance signals (e.g., Gemma-generated match scores), which are not independent ground truth.

\section{Problem Statement and Objectives}

Job seekers face a two-part problem. First, discovery: public job boards return results by simple keyword matching and provide no explanation of why a role is a good fit. Second, action: once a candidate identifies a role, preparing a tailored cover letter, adjusting their resume, and reaching out to a recruiter each require substantial manual effort. Generative AI is well-suited to both parts, but most course demonstrations address only one narrow slice, typically a single prompt or a chatbot without real data.

This project asks: \textit{Can we build a complete, working GenAI application that reads a resume, retrieves real job postings, explains match quality through a language model, and then supports every downstream application action with AI-generated content?}

The objectives, framed around what the course taught, were:

\begin{itemize}
  \item Design and implement a full RAG pipeline: corpus ingestion, vector embedding, Pinecone retrieval, candidate reranking, and Gemma-based generation.
  \item Use Gemma as the engine for five distinct AI tasks within a single application: job match explanation, cover letter writing, resume tailoring, keyword-gap analysis, and recruiter email drafting.
  \item Build a grounded system where the LLM is always conditioned on retrieved context, never generating jobs or qualifications from parametric memory alone.
  \item Add evaluation infrastructure to measure retrieval quality, generation faithfulness, output format consistency, diversity, and latency.
  \item Implement prompt safety: injection detection, resume quality gating, and output faithfulness checking.
\end{itemize}

\section{System Overview}

JobMatch AI is a browser-based application backed by a Python FastAPI server. The frontend is a static HTML/CSS/JavaScript interface with Google SSO authentication. The backend hosts the RAG pipeline, calls the Gemma API for all generative tasks, reads and writes user state to SQLite, and queries Pinecone for vector retrieval. External job data is ingested from a local Adzuna India dataset (4,000 records) and optionally from live job board APIs.

\section{The RAG Pipeline}

\subsection{Overview}

The RAG pipeline is the core of the system. Every job recommendation the user sees is the output of this pipeline, not a response generated purely from the model's parametric knowledge. The pipeline is designed so that Gemma is always conditioned on retrieved, real job postings from the indexed corpus. This prevents hallucination of non-existent roles and grounds the model's explanations in actual job descriptions.

The pipeline runs end-to-end on every recommendation request. Gemma generation is non-deterministic; retrieval and reranking are stable given the same inputs and embedding model, making the system debuggable and testable.

\subsection{Step 1: Resume Parsing and Quality Gating}

The pipeline begins when the user uploads a PDF resume. The backend uses \texttt{pdfplumber} to extract raw text, then applies a quality gate before proceeding. The gate checks whether the content resembles a real resume: it rejects documents that consist primarily of repeated instructions, contain no recognisable career content, or appear to be prompt-injection attacks disguised as resumes. PDFs that fail the gate are returned with an error; only valid resumes proceed to the query construction step. This prevents adversarial or malformed documents from influencing the retrieval or generation stages.

\subsection{Step 2: Query Canonicalisation}

The extracted resume text and any user-supplied constraints (desired role, location, salary, work type, skills) are normalised into a canonical query string. Role and skill inputs are lowercased, deduplicated, and joined into a structured format, for example: \texttt{Roles: data analyst | Skills: python, sql | Location: Mumbai}. Prompt-injection patterns are detected and stripped at this stage from all untrusted fields.

Canonicalisation ensures retrieval stability: paraphrased inputs that express the same job need produce the same canonical query and therefore retrieve the same set of candidates. This makes the system's behaviour predictable and testable.

\subsection{Step 3: Pinecone Vector Retrieval}

Each job posting in the corpus is embedded as a 1536-dimensional vector using OpenAI \texttt{text-embedding-3-small}. We embed a concatenation of structured job fields (role, company, location, skills, description), L2-normalise the result, and store it in Pinecone.

At query time, the canonical query is embedded using the same model and submitted to Pinecone for cosine-similarity search, returning the top-20 candidates. Because the embedding is semantic rather than token-hash based, retrieval is more robust to paraphrases and synonymy (e.g., ``ML engineer'' vs.\ ``machine learning engineer''), potentially improving retrieval quality beyond literal keyword overlap.

\subsection{Step 4: Keyword Reranking}

The top-20 Pinecone candidates are reranked using a deterministic scoring function that combines six signals: the original Pinecone cosine score, keyword overlap between the profile and the job description, role-title match, resume text alignment, location and work-type compatibility, and salary range fit. The reranker promotes jobs that are more aligned with the full user profile, beyond what raw cosine similarity captures. The top-10 reranked candidates are selected and passed to Gemma.

\subsection{Step 5: Gemma Generation}

The top-10 reranked job postings are assembled into a structured prompt along with the canonical query and relevant resume context. Gemma receives a system prompt that instructs it to act as a precise career advisor and return a structured markdown response listing five recommended jobs, each with:

\begin{itemize}
  \item A \textbf{Match Score} out of 10, reflecting how well the job fits the candidate's profile.
  \item A \textbf{Fit Rationale}: a concise explanation of why this role matches the candidate's skills, experience, and preferences.
  \item An \textbf{Apply Link} drawn directly from the retrieved job record.
\end{itemize}

Because the candidate list is passed explicitly as context, Gemma cannot recommend a job that was not retrieved. Faithfulness is enforced architecturally, not just by prompting. If the Gemma API call fails or returns a response the frontend cannot parse, the backend returns a deterministic fallback response so the application remains usable.

\subsection{Prompt Design and Safety}

The system prompt is carefully structured to produce consistent, parseable output. It specifies the expected markdown schema, instructs the model not to invent job details beyond what appears in the context, and explicitly disallows any preamble or explanatory text outside the defined format. This prompt engineering contributes to the Format Consistency score of 0.75 observed in evaluation: six of eight format rules pass at 100\%, with the remaining failures attributable to LLM formatting variability (e.g., Gemma occasionally folding the company name and location into the job heading rather than listing them as separate bold fields). Stricter parsing or schema enforcement would likely improve this metric.

Prompt injection safety is enforced at the canonicalisation stage and again at the generation stage. Patterns such as ``ignore previous instructions'' or ``disregard the above'' are detected and removed from all user-controlled inputs before they reach the prompt.

\section{AI-Powered Features}

Beyond the core RAG pipeline, JobMatch AI implements four additional Gemma-powered features. Each is designed as a grounded generation task: Gemma always receives the relevant job description and resume as context, ensuring that the output is specific to the candidate and the role rather than generic.

\subsection{Cover Letter Generation}

When a user selects a job from their results or saved applications, they can request a cover letter. The backend constructs a prompt containing the full job description, the user's resume text, and the extracted profile summary. Gemma is instructed to write a professional, concise cover letter that highlights the candidate's most relevant skills and experience in relation to the specific role and company. The output is grounded in the actual job description, so it references the employer's requirements rather than producing a generic template. The generated letter is displayed in a modal and can be emailed directly to the user via the Resend integration.

\subsection{Resume Tailoring and Keyword-Gap Analysis}

Each job posting has a different vocabulary, and applicant tracking systems often filter resumes before a human reads them. The resume tailoring feature addresses this by sending the job description and the candidate's resume to Gemma with a prompt that asks the model to:

\begin{enumerate}
  \item Identify skills and keywords present in the job description that are missing from the resume.
  \item Suggest specific rewordings or additions that would improve the resume's alignment with the role.
  \item Flag experience claims in the resume that could be strengthened given the job's requirements.
\end{enumerate}

The output is a structured tailoring report displayed in the interface, giving the candidate actionable, role-specific editing guidance. A companion keyword-gap endpoint returns only the present and missing keyword lists, useful for a quick comparison before applying.

\subsection{Recruiter Email Drafting}

Cold outreach to recruiters or hiring managers is often the most effective but most time-consuming part of a job search. The recruiter email feature gives Gemma the job description, the user's resume, and a brief description of the candidate's intent (e.g.\ expressing interest, requesting an informational interview). Gemma drafts a short, professional email that references the specific role and company and highlights the most relevant part of the candidate's background. The draft is returned to the user for review and editing before sending.

\subsection{Job Match Explanation}

The primary RAG response includes a fit rationale for each recommended job. This is not a simple template: Gemma reads the full job description and produces a paragraph explaining \textit{why} this role matches the candidate, citing specific skills, experience, or qualifications from the resume that align with the posting's requirements. Users consistently find this explanation more useful than a numeric score alone, as it identifies which aspects of their background are most relevant and where gaps exist.

\section{Data and Ingestion Pipeline}

The quality of RAG output depends directly on the quality of the indexed corpus. JobMatch AI ships with a local Adzuna India dataset of 4,000 JSONL records and supports optional live ingestion from seven external job board APIs (Adzuna, Jooble, Greenhouse, Lever, Remotive, Arbeitnow, USAJobs). Each posting is normalised into a common schema covering title, role category, company, location, salary range, skill tags, description, and source URL before embedding and indexing.

Three indexing modes are supported:

\begin{itemize}
  \item \texttt{csv\_only}: local dataset only; suitable for offline development.
  \item \texttt{live\_only}: live external APIs only; requires valid credentials.
  \item \texttt{hybrid}: combines both and deduplicates by job ID before indexing.
\end{itemize}

New jobs added through the admin panel are embedded and indexed immediately without a full rebuild. The force-reindex flag clears and rebuilds the Pinecone index from scratch when needed.

\section{Evaluation}

\subsection{Setup and Methodology}

19 evaluation records were collected from 10 distinct user profiles run through the live pipeline. Each record logs the canonical query, the pre- and post-rerank candidate lists (20 each), the complete Gemma output, retrieval latency, and generation latency, all stored in \texttt{eval\_log.jsonl}.

Standard RAG evaluation frameworks (e.g., RAGAS, DeepEval) were not applied because this task is a recommendation setting without ground-truth relevance judgments for retrieved items. Instead, we report proxy metrics based on signals available from the pipeline: (1) Gemma-generated match scores as graded relevance for NDCG (not independent ground truth, so this metric is partially self-referential); (2) Precision@K computed using a relative cosine-similarity threshold over the retrieved set (a proxy, not human-annotated relevance); and (3) generation-level checks such as faithfulness and format consistency. The evaluation is therefore preliminary due to both the small sample size (19 runs, 10 profiles) and the absence of labeled relevance data.\footnote{Recall@K, Hit@K, MRR, and MAP@K require relevance judgments or labeled data (e.g., human-annotated relevance) to compute reliably.}

\subsection{Discussion}

\textbf{Faithfulness is the most important result.} A score of 1.0 across all 19 runs provides strong evidence that the RAG architecture works as intended in this evaluation: Gemma never recommended a job outside the retrieved set. In a job-recommendation context this is critical, since a hallucinated posting would send a candidate to a non-existent application link.

\textbf{Retrieval quality is measured with proxies.} Precision@5 = 1.0 indicates that the top-5 retrieved candidates are consistently above the run-specific cosine threshold, but this is not equivalent to human relevance. NDCG@10 = 0.69 provides a coarse signal and should be interpreted cautiously because it uses Gemma-generated match scores as graded relevance (i.e., it is partially self-referential). Using semantic embeddings (\texttt{text-embedding-3-small}) can improve robustness to paraphrases and synonymy, but fully reliable ranking evaluation would require labeled relevance judgments.

\textbf{Reranker improves alignment and diversity, not NDCG at small K.} The ablation shows that reranking does not improve NDCG at K=3 or K=5 and has near-zero impact by K=10. This suggests the reranker's value is primarily in profile alignment and result diversity, rather than consistent gains in ranking quality under the proxy NDCG signal.

\textbf{Latency.} Generation (~30 s) dominates retrieval (~2 s). The high retrieval standard deviation is driven by Pinecone cold-start on the first query of a session.

\section{Limitations}

\textbf{Missing relevance labels.} The evaluation lacks human-annotated relevance judgments for retrieved jobs, so retrieval metrics rely on proxies (cosine thresholds and model-generated match scores) rather than ground-truth labels.

\textbf{No labeled feedback data.} Feedback-based metrics (Recall@K, MRR, MAP@K) require relevance judgments or labeled data, which were not collected during this evaluation run. The system has a feedback endpoint; populating it with even 20--30 labeled sessions would unlock a meaningful additional evaluation dimension.

\textbf{Self-evaluation bias in match scores.} Gemma assigns match scores to its own recommendations. A model may score its own output more favourably than an independent judge would. The LLM-as-judge metric (a separate judge prompt) would partially address this but was not run due to API cost.

\textbf{LLM document quality.} Cover letters, tailored resumes, and recruiter emails require user review before use. Gemma may produce plausible but imprecise claims if the resume context is sparse or ambiguous.

\section{Future Scope}

The highest-impact improvement is adding a small labeled evaluation set with human relevance judgments for retrieved jobs. This would enable standard ranking metrics (e.g., Recall@K, MRR, MAP@K) and allow proxy thresholds to be calibrated more rigorously. On the retrieval side, hybrid retrieval and query expansion can further improve recall and diversity even with semantic embeddings (\texttt{text-embedding-3-small}).

Additional improvements worth pursuing:

\begin{itemize}
  \item \textbf{Hybrid retrieval:} combine Pinecone vector search with BM25 keyword ranking for complementary coverage.
  \item \textbf{Feedback loop:} collect relevance labels (e.g., user ratings) and use them for personalised reranking and ongoing evaluation.
  \item \textbf{Agentic resume editing:} allow Gemma to iteratively refine the resume against a job description over multiple turns rather than producing a one-shot tailoring report.
  \item \textbf{Interview preparation:} generate likely interview questions based on the job description and the candidate's resume gaps.
  \item \textbf{ATS formatting:} produce downloadable, ATS-compatible resume exports alongside the tailoring analysis.
\end{itemize}

\section{Conclusion}

JobMatch AI is a grounded, full-stack generative AI application built around a RAG pipeline. The system uses Gemma for five distinct AI tasks, all conditioned on retrieved job data or the candidate's resume: job match explanation, cover letter generation, resume tailoring, keyword-gap analysis, and recruiter email drafting. The RAG architecture ensures that every recommended job exists in the indexed corpus, confirmed by a Faithfulness score of 1.0 across all 19 evaluation runs.

Retrieval uses semantic embeddings (\texttt{text-embedding-3-small}) and the end-to-end evaluation suggests reasonable retrieval behaviour under proxy metrics. Precision@5 = 1.0 indicates strong performance relative to the run-specific cosine threshold, while NDCG@10 = 0.69 provides a coarse, partially self-referential signal because it is based on Gemma-generated match scores rather than independent relevance labels.

The project demonstrates that RAG is not just a retrieval trick: when combined with a capable language model and thoughtful prompt design, it enables a family of grounded generative features that would be impossible, or unsafe, without the retrieved context.
